# 1. 에이전트 유형

### 반사형 에이전트

- 입력과 행동을 직접 연결하며 내부 추론 과정을 거치지 않음
- 조건이 참이면 도구 실행 → 사전에 정의된 트리거가 감지되면 즉시 적절한 도구 호출
- 중간 추론 단계 생략 → 매우 짧은 지연시간, 예측 가능한 성능

### 리액트 에이전트

- 추론과 행동을 반복 루프 내에서 교차 수행
- 사고를 생성하고 도구를 선택해 호출한 뒤 결과를 관찰하고 다시 계획을 갱신
- 복잡한 작업을 여러 관리 가능한 단계로 분할 및 중간 결과 반영과 계획 조정 가능
- 탐색적 시나리오, 유연성, 디버깅과 감사에 유리

### 계획 후 실행 에이전트

- 다단계 계획을 생성하는 계획 단계 → 장기적인 추론 집중
- 계획된 각 단계를 도구 호출을 통해 수행하는 실행 단계 → 필요한 툴만 호출
- 명확한 분해, 디버깅 용이성, 비용 효율성

### 쿼리 분해 에이전트

- 복잡한 질문을 반복적으로 하위 질문으로 나누고 각 하위 질문마다 검색 및 다른 도구 호출 후 최종 답변 생성
- 자문 후 검색 : 어떤 후속 질문이 더 필요한가 → 검색 호출 → 다음 질문은 무엇인가 → … → 최종 답은 무엇인가

### 성찰형 에이전트

- 생각화 행동을 교차시키는 것에 그치지 않고 다음 단계로 진행하기 전에 과거 단계를 검토해 실수를 찾아내고 수정
- 리액트 패러다임 확장
- 리플랙트 프레임워크 : 목표 상태에 대한 성찰을 바탕으로 자신의 추론을 지속적으로 정렬
- 모델이 자신의 추론 과정을 비판적으로 검토하고 논리적 오류를 수정하며 성공적인 전략을 강화하도록 유도

### 심층 리서치 에이전트

- 개방형이고 매우 복잡한 조사 과제를 전문적으로 다룸
- 광범위한 외부 지식 수집, 가설 검증, 통합이 필요한 작업
- 전체 연구 아젠다 계획 → 각 하위 토픽을 자문 기법을 통해 구체적인 쿼리로 분해 → 학술 검색 API부터 도메인 특화 데이터베이스 등 다양한 도구 호출 및 결과의 관련성 및 신뢰성 성찰 → 각 단계에서 LLM 기반 요약과 비평을 활용해 점진적으로 발전하는 보고서나 추천안으로 통합
- 역량, 적응성, 투명성
- 높은 비용, 지연시간, 취약성

# 2. 도구 선택

### 표준 도구 선택

- 도구와 그 정의, 설명을 파운데이션 모델에 함께 제공
- 주어진 컨텍스트에 가장 적합한 도구를 선택하도록 모델에 요청
- 도구 모음과 비교되고 그중 가장 가까운 도구로 모델 출력
- 도구에 간결하고 설명적인 이름 붙이기
- 도구의 고유 목적을 한 문장으로 요약
- 설명에 예시 호출을 포함해 전형적인 입력과 출력 형태 작성

### 시맨틱 도구 선택

- 모든 사용 가능한 도구를 시맨틱 표현으로 인덱싱하고 시맨틱 검색으로 가장 관련성 높은 도구를 가져옴
- 선택해야 할 도구 수 감소 및 작은 후보 집합 안에서 파운데이션 모델이 올바른 도구와 파라미터 선택 가능
- 벡터 데이터베이스에 인덱싱 → 가장 관련성 높은 상위 도구 검색

### 계층적 도구 선택

- 시나리오에 포함된 도구 수가 매우 많을 때 유용
- 도구 중 상당수가 시맨틱하게 서로 비슷하고 더 높은 지연시간과 복잡성을 감수하더라도 도구 선택 정확도를 개선하고자 할 때 사용
- 도구를 여러 그룹으로 조직하고 각 그룹에 대한 설명 제공

# 3. 도구 실행

- 파라미터화 : 언어 모델에서 도구 실행을 안내할 파라미터를 정의하고 설정하는 과정
- 모델이 작업을 어떻게 해석하고 특정 요구사항을 충족하도록 응답을 어떻게 조정할지를 결정

# 4. 도구 토폴로지

### 단일 도구 실행

- 작업을 해결하는 데 가장 적합한 단일 도구를 선택
- 도구 정의에 따라 올바르게 파라미터 설정 → 실행 및 출력값을 사용해 사용자에게 제공할 최종 응답 구성

### 병렬 도구 실행

- 여러 도구 병렬 실행 시 복잡성 증가
- 시맨틱 도구 선택을 사용해 실행될 수 있는 도구의 최대 개수 호출
- 어떤 도구들이 이미 선택되었는지에 대한 추가 컨텍스트를 함께 제공 → 도구를 추가하지 않기로 결정할 때까지 계속 도구 추가

### 체인

- 일련의 작업을 순차적으로 실행하는 패턴
- 각 작업은 이전 작업이 성공적으로 완료되어야만 이어서 수행 가능
- 특정 목표를 달성하기 위해 어떤 작업을 어떤 순서로 수행해야 하는지 결정하고 각 작업이 끊김 없이 다음 작업으로 이어지도록 보장하는 과정
- 랭체인 : 직접 Chain 객체로 구성하지 않고 기존 Runnable들을 합성하는 방식으로 체인 구축(LCEL)

### 그래프

- 여러 의사결정 지점을 포함하는 지원 시나리오에 유리
- 조건부 엣지와 병합 엣지 모두 정의 가능
- 각 노드 : 개별 도구 호출
- `add_conditional_edges` : 에이전트가 어떤 조건에서 단계 간을 전이할 수 있는지 명시

# 5. 컨텍스트 엔지니어링

- 오케스트레이션의 핵심 구성 요소
- 에이전트 계획의 각 단계가 효과적으로 작동하는 데 필요한 올바른 정보와 지침을 갖추도록 보장
- 사용자 메시지, 검색된 지식, 워크플로 상태, 시스템 프롬프트 등 모든 입력을 동적으로 조합해 작업 성능을 최대화하는 구조적이고 토큰 효율적인 컨텍스트 윈도우를 만드는 작업
- 어떤 정보를 포함할지 최대한 명료하고 관련성 있게 보이도록 어떻게 구조화할지, 토큰 한도 안에 얼마나 효율적으로 담을지를 결정하는 일
- 메모리나 지식 베이스에서 큰 텍스트 덩어리를 무작정 붙이기보다 가장 유용한 정보만 검색하고 선택해 관련성 우선시 하기
- MCP 같은 구조화된 포맷이나 스키마를 사용해 상태와 검색된 지식을 예측 가능하고 해석 가능한 방식으로 모델에 전달함으로써 명료성 유지 필요
- 히스토리가 길어지면 요약을 적용해 중요한 핵심 정보를 보존하고 불필요한 표현은 제거하여 토큰 사용량을 최적화해야 함
- 각 추론 스텝마다 컨텍스트가 에이전트의 현재 목표, 워크플로 단계, 사용자 입력을 반영하도록 동적으로 조립되도록 해야 함
