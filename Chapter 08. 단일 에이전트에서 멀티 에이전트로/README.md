# 1. 에이전트는 몇 개나 필요할까

### 단일 에이전트 시나리오

- 난이도가 그리 높지 않은 작업, 제한된 수의 도구, 복잡성이 낮은 환경에 적합
- 단순성, 낮은 리소스 요구사항, 지연시간
- 도구와 책임의 개수가 늘어날 때 핵심 병목 증가

### 멀티 에이전트 시나리오

- 여러 에이전트가 공통 목표를 달성하기 위해 협력
- 작업이 복잡하고 다양한 도구 모음, 병렬 처리, 동적인 환경에 대한 적응력이 필요한 문제에 유리
- 도구를 그룹화한 후, 각 전문가를 위한 별도의 언어 모델 인스턴스에 바인딩

### 스웜

- 많은 수의 단순한 에이전트가 개별적으로는 지능이 거의 없지만, 국소적인 상호작용과 단순한 규칙을 통해 집단적으로 지능적인 창발적 행동 발현
- 탈중앙화와 자기 조직화 강조
- 확장성, 견고성, 유연성, 분산 문제 해결

# 2. 에이전트 추가 원칙

- 작업 분해 : 복잡한 작업을 더 작고 관리 가능한 하위 작업으로 나누는 것의 중요성 강조
- 전문화 : 에이전트가 자신의 강점에 부합하는 역할을 맡도록 해 시스템의 집합적 역량 극대화
- 파시모니 : 원하는 기능과 성능을 달성하는 데 필요한 최소한의 에이전트만 추가하도록 권장하는 원칙
- 조율 : 에이전트 간 정렬 상태를 유지하려면 견고한 통신 프로토콜을 구축해 정보 공유를 효율적으로 하고 충돌 위험을 줄여야 함
- 견고성 : 장애 허용과 탄력성 증가
- 효율성 : 에이전트를 추가하는 것과 그로 인해 발생하는 잠재적 복잡성과 리소스 요구 사이의 트레이드오프를 평가하는 기준

# 3. 멀티 에이전트 조율

### 민주적 조율

- 시스템 내 모든 에이전트가 각자 동등한 의사결정 권한을 가지며 행동과 솔루션에 대해 합의에 도달하는 것
- 어떤 에이전트도 리더로 지정되지 않는 탈중앙화된 제어

### 관리자 중심 조율

- 더 중앙집중적인 접근 방식을 취하며 하나 이상의 에이전트를 관리자로 지정해 하위 에이전트의 행동을 감독하고 지시하는 책임을 부여
- 의사결정 과정 간소화, 시스템 효율성 증가

### 계층형 조율

- 구조화된 계층을 통해 중앙집중식 제어와 탈중앙화 제어 요소를 결합하는 다단계 조직 방식
- 상위 수준 에이전트 → 하위 수준 에이전트 감독 및 지시, 일정 수준의 자율성 부여
- 작업을 서로 다른 수준에서 관리 가능

### 액터-크리틱 접근법

- 평가 주도 반복을 가볍게 적용한 형태
- 액터는 답변, 계획, 행동과 같은 후보 출력을 생성하는 역할, 크리틱은 사전에 정의된 평가 기준에 따라 출력을 받아들이거나 거부하는 품질 게이트 역할
- 원하는 품질 임계값에 도달할 때까지 계속해서 후보를 만들어 내고 판정

# 4. 에이전틱 시스템의 자동 설계(ADAS)

- 수작업으로 만든 아키텍처에서 벗어나, 스스로를 설계하고 평가하며 반복적으로 개선할 수 있는 시스템으로 나아가는 에이전트 개발의 변혁적 접근
- 에이전트가 완전히 새로운 구조와 모듈을 스스로 발명할 수 있게 함으로써 기존 접근을 넘어서는 것이 목표
- 코드를 통해 에이전트를 정의함
- MAS 알고리즘 : 메타 에이전트가 어떻게 에이전트 시스템을 자율적으로 생성하고 개선할 수 있는지를 보여줌

# 5. 에이전트 통신 기법

### 로컬 통신과 분산 통신

- 에이전트가 서비스, 컨테이너, 노드에 걸쳐 분산되는 즉시, 통신 방식은 명시적이고 비동기적이며 장애 허용성을 갖춰야 함
- 오토젠 같은 프레임워크가 메모리 내 라우터를 사용하여 에이전트 간 메시지 전달과 도구 호출 조율

### A2A 프로토콜

- 자율 에이전트들이 협력하여 더 복잡한 목표를 달성할 수 있도록 돕는 유망한 시도
- 에이전트가 내부 로직이나 구현 세부 사항을 노출하지 않고도, 서로 탐색하고 협업을 조율하며 구조화된 요청을 교환할 수 있는 표준화된 크로스 플랫폼 메커니즘 제공
- 에이전트 카드 : 각 에이전트가 자신의 신원, 기능, 엔드포인트, 지원 인증 방식을 알리기 위해 발행하는 기계 판독 가능한 JSON 기술서

# 6. 메시지 브로커와 이벤트 버스

- 송신자와 수신자의 결합을 분리하고 에이전트들이 공유 통신 패브릭을 통해 비동기적으로 상호 작용 가능
- 느슨하게 결합된 멀티 에이전트 아키텍처에서 확장 가능하고 장애 허용성이 있으며 관측 가능한 워크플로 확립 가능
- 아파치 카프카 : 에이전트가 구조화된 이벤트를 발행 및 소비해야 하는 시스템에 적합한 고처리량 분산 이벤트 스트리밍 플랫폼
- 레디스 스트림과 래빗MQ : 더 낮은 처리량이나 단순한 사용 사례에 적합한 경량 대안, 지연시간이 낮고 배포가 쉬움
- NATS : 저지연, 고처리량 통신을 위해 설계된 경량 클라우드 네이티브 메시징 시스템

# 7. 액터 프레임워크

- 메시징과 연산을 하나의 통합된 모델로 제공
- 액터는 단순히 메시지를 교환하는 데 그치지 않고 자체 상태와 기능을 캡슐화함
- 정교한 분산 처리, 복원력, 동적 확장이 필요한 상황에서 우수

# 8. 오케스트레이션 및 워크플로 엔진

- 작업 순서를 정하고 재시도를 처리하며 의존성을 추적하고 에이전트 전반의 실패를 관리하는 로직
- 더 높은 수준의 추상화를 제공해, 복잡한 에이전틱 시스템에서 지속성, 복구 가능성 보장

# 9. 상태와 영속성 관리

- 여러 번의 실행, 워크플로, 시스템 재시작에 걸쳐 존재하는 공유 상태, 에이전트 메모리, 작업별 메타데이터 함께 관리
- 상태 저장 데이터베이스, 벡터 스토어, 오브젝트 스토리지, 상태 저장 오케스트레이션 프레임워크
