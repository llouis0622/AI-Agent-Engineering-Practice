# 1. 에이전트 시스템의 핵심 구성요소

- 에이전트의 능력, 효율, 적응성 결정
- 적절한 모델 선택, 도구, 메모리, 계획 능력 부여 등

# 2. 모델 선택

- 에이전트의 의사결정, 상호작용, 학습 역량 결정
- 에이전트가 입력을 해석하고 출력을 생성하며 환경에 적응하는 방식 결정 → 시스템의 성능, 확장성, 지연시간, 비용에 직접적인 영향
- 작업 복잡도 평가
- 모달리티 : 텍스트, 이미지, 오디오, 구조화 데이터 등 처리
- 개방성과 커스터마이징 가능성
- 범용 사전학습 모델 vs 맞춤형 모델
- 비용과 지연시간

# 3. 도구

- 에이전트가 특정 작업을 수행하고 문제를 해결할 수 있게함
- 에이전트의 기능적 빌딩 블록 → 작업 실행, 사용자 및 다른 시스템과의 상호작용 가능

### 특정 작업을 해결하는 도구 설계

- 로컬 도구 : 에이전트가 외부 의존성 없이 내부 로직과 계산만으로 작업 수행. 규칙 기반, 사전에 정의된 함수를 실행하는 방식. 수학 계산, 로컬 데이터베이스 조회, 미리 정한 규칙에 따른 간단한 의사결정 등
- API 기반 도구 : 에이전트가 외부 서비스나 데이터 소스와 상호작용. 실시간 데이터를 조회하거나 외부 시스템을 활용해 로컬 환경 너머의 기능 수행
- MCP : 표준 스키마를 통해 사용자 프로필, 대화 이력, 세계 상태, 작업 메타데이터 같은 구조화된 실시간 컨텍스트를 프롬프트에 직접 전달. 표준 스키마를 통해 구조화된 실시간 컨텍스트 제공, 도구 호출을 표준화된 방식으로 지원. 불필요한 도구 사용을 줄이고 대화 상태를 보존하며 실시간 상황 인식을 모델에 주입 시 효과적

### 도구 통합과 모듈성

- 필요에 따라 쉽게 통합하거나 교체할 수 있도록 독립적인 모듈로 설계
- 전체 시스템을 다시 구축하지 않고도 에이전트의 기능 확장 및 업데이트 가능

# 4. 메모리

- 에이전트가 정보를 저장하고 검색할 수 있게 하는 핵심 구성 요소
- 컨텍스트를 유지하고 과거 상호작용에서 학습하며 시간이 지남에 따라 더 나은 의사결정 가능

### 단기 메모리

- 에이전트가 현재 작업이나 대화와 관련된 정보를 저장하고 관리하는 능력
- 상호작용 중 컨텍스트를 유지하는 데 사용
- 에이전트가 실시간으로 일관된 의사결정을 내릴 수 있도록 도움
- 롤링 컨텍스트 윈도우 : 최근 정보를 일정 범위 내에서 계속 갱신하면서 오래된 데이터를 버릴 수 있음

### 장기 메모리

- 에이전트가 오랜 기간에 걸쳐 지식과 경험을 저장하도록 하여 과거 정보를 바탕으로 향후 행동을 결정할 수 있도록 함
- 데이터베이스, 지식 그래프, 파인튜닝된 모델을 통해 구현
- 에이전트가 사용자 선호도나 과거 성능 지표 같은 구조화된 데이터를 저장하고 필요할 때 검색 가능

### 메모리 관리 및 검색

- 저장된 데이터를 체계적으로 구성하고 인덱싱해 필요할 때 쉽게 검색할 수 있도록 하는 것
- 오래되거나 불필요한 세부 정보로 메모리에 불필요한 정보가 쌓이는 것을 방지하기 위해 일부 정보 휘발 필요

# 5. 오케스트레이션

- 개별적인 기능을 end-to-end 솔루션으로 전환
- 여러 스킬을 배치하고 상황에 따라 실행하며 전체 과정을 감독하는 논리로 각 단계가 다음 단계로 자연스럽게 이어지며 명확한 목표를 이루도록 함
- 도구나 스킬 호출의 가능한 순서를 평가하고 그 결과를 예측하며 여러 단계를 거치는 작업에서 가장 성공 가능성이 높은 경로를 선택
- 진행 상황과 환경을 지속적으로 모니터링하면서 필요에 따라 워크플로를 일시 중단하거나 경로를 재조정해 목표에서 벗어나지 않아야 함

# 6. 설계 트레이드오프

### 성능 : 속도와 정확도의 균형

- 높은 성능 → 정보를 빠르게 처리하고 의사결정을 내리며 작업 수행 가능, 정밀도 떨어짐
- 높은 정확도 → 복잡한 모델이나 계산 집약적인 기법 필요, 전체 속도 감소
- 하이브리드 전략 : 에이전트가 먼저 빠르고 대략적인 결과를 제시한 뒤 추가로 시간과 데이터를 활용해 이를 정교하게 보완하는 방식

### 확장성 : 에이전트 시스템의 엔지니어링적 확장

- 시스템이 커질수록 데이터량, 동시 처리 작업 수, 연산 리소스의 효율적인 관리
- GPU → 대규모 AI 모델의 학습과 추론을 가속화하는 기반, 효율적인 확장을 위해 병목 현상, 자원 낭비, 비용 상승을 방지하는 정교한 엔지니어링 필요
- 동적 GPU 할당 : 실시간 수요에 따라 GPU를 배정해 GPU를 고정적으로 할당하는 대신 필요할 때만 사용해 유휴 시간을 줄이고 활용도 극대화
- 탄력적 GPU 프로비저닝 : 클라우드나 온프레미스 GPU 클러스터를 이용해 현재 워크로드에 따라 자동으로 리소스를 확장하거나 축소
- 유선순위 큐잉 + 지능형 작업 스케줄링 : 중요한 작업은 즉시 GPU 접근 권한을 부여하고 덜 중요한 작업은 피크 시간대에 대기시켜 효율성 증가
- 비동기 작업 실행 : 이전 작업의 완료를 기다리지 않고 GPU 작업을 병렬로 처리해 GPU 자원 활용도를 극대화하고 작업 간 유휴 시간 최소화
- 동적 로드 밸런싱 : 활용률이 낮은 GPU로 작업을 분산시켜 특정 GPU에 병목이 생기지 않도록 함
- 수평 확장 : 시스템의 처리 능력을 높이기 위해 GPU 노드를 추가하는 방식
- 하이브리드 클라우드 방식 : 워크로드가 유동적인 시스템에 사용하여 확장성 증가
- 버스트 스케일링 : 클라우드 GPU에 일시적으로 작업을 분산시켜 처리 용량 확장

### 신뢰성 : 견고하고 일관된 에이전트

- 에이전트가 일정한 기간 동안 일관되고 정확하게 작업을 수행할 수 있는 능력
- 예상된 상황뿐 아니라 예기치 못한 조건에서도 문제 없이 작동해야 하며 이를 통해 사용자와 이해관계자의 높은 신뢰 확보
- 장애 허용 : 에이전트가 오류나 예기치 못한 사건을 감지하고 비정상 종료나 예측 불가능한 오작동 없이 이를 처리할 수 있도록 하는 것 → 네트워크 중단, 하드웨어 장애 등의 문제를 감지하고 정상 상태로 복구 필요
- 일관성과 견고성 : 에이전트가 이상적인 조건뿐 아니라 다양한 엣지케이스, 스트레스 테스트, 현실적 제약 조건에서도 안정적으로 작동하도록 해야 함
    - 철저한 테스트 : 단위 테스트, 통합 테스트, 실제 환경을 시뮬레이션한 테스트 등 다양한 검증 과정 필요. 엣지케이스, 예상치 못한 입력, 적대적 조건 등을 포괄해야 하며 이를 통해 에이전트가 다양한 환경에서도 안정적으로 작동함을 확인
    - 모니터링과 피드백 루프 : 신뢰할 수 있는 에이전트는 운영 환경에서 지속적인 모니터링을 통해 이상 징후를 탐지하고 환경 변화에 맞춰 작동 방식 조정 필요

### 비용 : 성능과 지출의 균형

- 에이전트를 개발, 배포, 유지하는 데 드는 비용 → 기대되는 이익과 투자 수익률(ROI)을 기준으로 평가
- 개발 비용 : 신뢰성과 확장성을 갖춘 에이전트 시스템을 구축하기 위해 방대한 테스트 인프라 필요
- 운영 비용 : 방대한 데이터를 처리하거나 대규모 메모리 유지, 버그 수정 및 시스템 개선
- 비용 대비 가치 : 시스템의 전체 목표와 예상 수명을 고려해 이루어짐
    - 경량 모델 : 특정 작업에서 규칙 기반 시스템이 딥러닝 모델과 유사한 성능을 낼 수 있다면 단순한 접근이 훨씬 비용 효율적
    - 클라우드 기반 리소스 : 클라우드 컴퓨팅 자원을 활용하면 초기 인프라 구축 비용을 줄이고 사용량 기반 과금 모델을 통해 확장성과 유연성 확보 가능
    - 오픈소스 모델과 도구 : 오픈소스 머신러닝 라이브러리와 프레임워크를 활용하면 소프트웨어 개발 비용을 줄이면서도 높은 품질의 에이전트 구현 가능

# 7. 아키텍처 디자인 패턴

### 단일 에이전트 아키텍처

- 가장 단순하고 직관적인 설계 형태
- 하나의 에이전트가 시스템 내 모든 작업을 관리하고 실행
- 환경과 직접 상호작용하며 다른 에이전트의 도움 없이 스스로 의사결정, 계획 수립, 실행 수행
- 명확히 좁은 범위의 작업에 적합, 하나의 엔티티로 처리할 수 있는 업무에 이상적
- 문제 도메인이 명확하고 작업이 단순하며 대규모 확장이 필요하지 않은 환경에서 효과적

### 멀티 에이전트 아키텍처 : 협업, 병렬성, 조율

- 여러 에이전트가 공동의 목표를 달성하기 위해 협력
- 작업의 성격에 따라 독립적으로, 병렬적으로, 혹은 조율된 방식으로 작동 가능
- 작업의 여러 측면을 전문화된 에이전트가 분담하거나 병렬 처리를 통해 효율성과 확장성을 높일 수 있는 상황에 적합
- 장점
    - 협업과 전문화 : 각 에이전트는 특정 작업이나 영역에 특화되도록 설계 가능
    - 병렬성 : 여러 작업 동시에 수행 가능
    - 확장성 향상 : 시스템이 성장함에 따라 새로운 에이전트를 추가해 더 많은 작업을 처리하거나 부하 분산 가능
    - 중복성과 복원력 : 여러 에이전트가 독립적으로 작동하므로 에이전트 일부가 작업에 실패하더라도 전체 시스템이 중단되지 않음
- 단점
    - 조율과 통신의 복잡성 : 에이전트 간 통신을 효율적으로 관리해야 함. 각 에이전트가 중복 작업, 충돌, 자원 경쟁을 피하기 위해 정보를 교환하고 작동 방식 조율 필요
    - 복잡성 증가 : 설계, 개발, 유지보수 훨씬 어려움
    - 효율성 저하 : 토큰 소비가 많아 효율이 낮아질 수 있음. 단일 에이전트보다 처리 자원과 계산 비용이 더 많이 듬

# 8. 모범 사례

### 점진적 설계

- 에이전트 개발의 핵심 접근법
- 초기부터 완벽한 결과물을 목표로 하기보다 작동하는 작은 프로토타입을 만들고 피드백을 반영해 여러 차례에 걸쳐 개선
- 문제의 조기 발견 : 초기 프로토타입을 배포하면 설계 결함이나 성능 병목을 시스템에 깊이 뿌리내기기 전에 찾아낼 수 있음
- 사용자 중심 설계 : 이해관계자와 최종 사용자, 개발자에서 빈번한 피드백 획득 가능
- 확장성 : 최소기능제품(MVP)이나 기초 에이전트로 시작 시 관리 가능한 단위로 점진적 성장 진행 가능
- 프로토타입 빠르게 개발 → 초기에 핵심 기능 구현에 집중
- 테스트하고 피드백 수집 → 개선을 진행할 때마다 사용자, 개발자, 이해관계자의 피드백 수용
- 개선 및 반복 → 피드백과 성능 데이터를 토대로 필요한 변경을 적용하고 다음 반복에서 시스템 수정

### 평가 전략

- 견고한 평가는 에이전트가 실제 환경을 처리하고 다양한 조건에서 작동하며 성능 기대치를 충족하는지 확인
- 정확성 : 설계에 기반에 에이전트가 일관되게 정확하고 기대한 출력을 제공하는지 확인
- 경계 테스트 : 매우 큰 데이터셋, 이례적 질의, 모호한 지침과 같은 엣지 케이스를 에이전트가 어떻게 처리하는지 평가
- 작업별 지표 : 해당 도메인의 정확도와 컴플라이언스 요구사항을 충족하는지 확인
- 사용자 만족도 점수 : NPS나 고객만족도(CSAT) 같은 지표로 사용자 상호작용에 대한 만족도 측정
- 작업 완료율 : 사용자가 에이전트의 도움으로 작업을 성공적으로 완료하는 비율 측정
- 명시적 신호 : 사용자가 피드백을 제공할 기회 생성
- 암묵적 신호 : 사용자-에이전트 상호작용을 분석해 오해, 지연, 감정, 부적절한 응답 등 공통 문제 지점 식별
- human-in-the-loop : 자동화된 평가와 인간의 판단을 결합해 에이전트 성능이 실제 표준과 일치하도록 함

### 실환경 테스트

- 에이전트를 실제 운영 환경에 배포하고 현실 조건에서의 작동 방식을 관찰하는 과정
- 실환경의 복잡성 노출 : 다양한 사용자, 엣지 케이스, 예기치 못한 과제에 대해 테스트
- 엣지 케이스 발굴 : 테스트 단계에서 고려하지 못했던 엣지 케이스 확보
- 부하 상태에서의 성능 평가 : 높은 작업량이나 사용자 수요 증가 상황에서 에이전트가 어떻게 작동하는지 관찰 가능
- 단계적 배포 : 제한된 환경에서의 소규모 테스트로 시작해 전체 배포로 확장
- 에이전트 작동 방식 모니터링 : 실환경 테스트 동안 에이전트의 작동 방식, 응답, 성능 지표 추적. 응답 시간, 정확도, 사용자 만족도, 시스템 안정성과 같은 KPI에 집중
- 사용자 피드백 수집 : 사용자와 소통해 상호작용 경험에 대한 피드백 수집
- 인사이트에 따른 반복 : 실환경 테스트에서 얻은 인사이트를 개발 주기로 되돌림
